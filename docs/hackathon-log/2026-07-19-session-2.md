## Work completed

### Runtime-compilation review and corrections

A high-effort review separated the runtime path into explicit responsibilities:

1. `RuleSetDefinitionSource` loads an immutable rule-set definition.
2. `RuleSetDefinitionValidator` validates structure, supported shapes, stage and
   position ordering, safe identifiers, runtime unit identities, and activity
   additions.
3. `TraditionalDrlRenderer` assembles traditional DRL from structured data and
   stored templates.
4. `RuntimeRuleSetCompiler` compiles an already-rendered rule set with the public
   KIE builder API and discovers the generated Rule Units.
5. `CompiledRuleSet` owns the resulting KIE container, generated unit references,
   release identifier, and repository-module cleanup.
6. A `RuleUnitInstance` is created only for execution and is always closed by the
   caller.

The review identified that runtime-generated units could not safely reuse the
same Java identities as Maven-generated units. Dedicated unit-data classes were
introduced under `org.acme.ruleunits.runtime.ra1`, `.ra2`, and `.ra3`, and
validation now requires the corresponding fully qualified class for each stage.
Runtime compilations also receive unique KIE release identifiers, allowing
successive versions to coexist in isolated KIE class loaders while an older
version drains.

The runtime path requires `drools-wiring-dynamic`. It was added with `runtime`
scope, which is the narrowest scope needed: application source does not compile
against its API, but its service provider must be present when KIE defines
executable-model classes that did not exist during Maven compilation.

KIE ownership was made explicit. Successful compilation returns a
`CompiledRuleSet` that retains the `KieContainer` and registered KIE module.
Closing it disposes the container, clears owned generated-unit references, and
removes the module from the KIE repository. Compilation or unit-discovery failure
performs the same cleanup before propagating the failure. The implementation does
not claim immediate JVM class unloading; it claims only release of owned strong
references and KIE resources.

### Snapshot publication and refresh

Increment 12 added atomic last-known-good snapshot management:

- `RuleSetSnapshotManager` holds the current snapshot in an atomic reference and
  serializes publication and shutdown.
- `RuleSetLease` gives each reader a reference-counted lease.
- Publication atomically installs the candidate and retires the previous slot.
- A retired slot rejects new leases but remains usable by leases already in
  flight.
- The retired runtime closes exactly once after its last lease drains.
- Manager shutdown follows the same retirement and drain policy.
- Only the snapshot owner closes a published `CompiledRuleSet`.

Increment 13 added `RuleSetRefreshCoordinator`. A fair lock serializes the entire
refresh operation; no second refresh can enter loading while another is in any
of these phases:

`LOAD -> VALIDATE -> ASSEMBLE -> COMPILE -> PUBLISH`

Validation therefore always precedes DRL assembly and compilation. A candidate
is not transferred to the snapshot manager until every preceding phase succeeds.
Before the first valid publication, rule execution is unavailable. After a valid
publication, a failure in any later refresh leaves the last-known-good snapshot
unchanged and executable.

Refresh results and ordinary logs contain the rule-set name, attempted version
when known, failure phase, correlation ID, exception type, and a fixed sanitized
summary. They do not retain the underlying exception message, full generated DRL,
embedded stack traces, or verbose KIE diagnostics. Spring Boot configuration
also disables the narrow internal Drools `KieProject` logger that otherwise logs
compiler details before the coordinator can sanitize them.

### Dynamic execution

Increment 14 added `DynamicRulesOrchestrator` and the `WorkOrderRulesEngine`
abstraction. One lease is acquired for the complete work-order execution. RA1,
RA2, and RA3 are executed in order, with the existing save boundary after each
stage, and every `RuleUnitInstance` is closed with try-with-resources.

Using one lease across all three stages is essential. If a new rule-set version
is published while a work order is being saved after RA1, that work order still
uses its leased version for RA2 and RA3. New work may immediately acquire the new
snapshot. The retired snapshot and its KIE module remain alive until the in-flight
work order releases its lease.

The six agreed characterization behaviors were rerun through actual dynamically
compiled units and the mocked repository boundary:

- duplicate original occurrences each produce a replacement, and later
  deactivation attribution identifies the rule that made the change;
- category deactivation and later refinement preserve rule attribution;
- corrected RA3 matching requires an active CAT3 occurrence after RA2;
- a valid nonmatching work order still crosses every save boundary;
- a missing requested work order is ignored;
- an invalid RA2 activity addition leaves the work order at its successfully
  saved RA1 state and prevents the failing stage from being saved.

## Important technical findings

### `drools-wiring-dynamic`

- **Observed failure or risk:** Runtime executable-model classes cannot be
  defined by static wiring merely because build-time Rule Units compile.
- **Root cause:** Database-assembled DRL is compiled after Maven has finished;
  its generated classes do not exist in the build output and must be defined in
  the runtime KIE class loader.
- **Chosen solution:** Add `drools-wiring-dynamic` with `runtime` scope.
- **Verification:** The runtime compiler tests compile, discover, instantiate,
  and fire RA1, RA2, RA3, and traditional `accumulate` units. Removing the need
  for dynamic wiring was not supported by the experiment.

### Build-time versus runtime-generated Rule Units

- **Observed failure or risk:** A successful Maven `kjar` build proves
  Maven-generated units, not the database-driven runtime path. Runtime tests
  could accidentally discover checked-in generated units and appear to pass.
- **Root cause:** Build-time and runtime units originally shared fully qualified
  unit-data identities, and class loading is parent-first.
- **Chosen solution:** Keep the reference units generated during Maven build, but
  use dedicated `org.acme.ruleunits.runtime.*` data classes and require those
  exact identities in stored stage definitions.
- **Verification:** Tests execute consequences unique to the runtime-rendered
  rules, reject missing generated runtime units, and validate the required stage
  identities.

### Generated class and package collisions

- **Observed failure or risk:** Reusing names can select the build-time class or
  allow one runtime refresh to interfere with another.
- **Root cause:** Generated service metadata identifies units by their data class,
  while KIE modules also need distinct release identities.
- **Chosen solution:** Separate build-time and runtime packages, validate unique
  stage unit classes, use a unique release identifier for every compilation, and
  retain each version's container for the lifetime of its snapshot.
- **Verification:** Successive compilations have distinct release identifiers;
  closing one does not prevent the other from executing. A refresh during work
  proves old and new generated versions coexist until lease drain.

### Resource and KIE repository cleanup

- **Observed failure or risk:** Repeated refresh could retain KIE modules,
  containers, generated-unit references, and class loaders, including when
  compilation or discovery fails.
- **Root cause:** KIE builder output is registered in the process-wide KIE
  repository, and disposing a container alone does not express the full module
  ownership boundary.
- **Chosen solution:** `CompiledRuleSet` owns container disposal and repository
  removal; compiler failure cleans partial resources; snapshot retirement invokes
  that close exactly once after drain.
- **Verification:** Focused tests assert module presence while usable, module
  removal after close, idempotent close, cleanup after retirement, and continued
  execution of an independent compilation. This verifies owned-reference cleanup,
  not immediate JVM class unloading or long-duration heap behavior.

### Oracle `TIMESTAMP(6)` and `LocalDateTime`

- **Observed failure or risk:** Treating validation and activation timestamps as
  instants would introduce timezone semantics absent from Oracle `TIMESTAMP(6)`.
- **Root cause:** Oracle `TIMESTAMP(6)` stores a local date-time without timezone.
- **Chosen solution:** Map `VALIDATED_AT` and `ACTIVATED_AT` to Java
  `LocalDateTime`.
- **Verification:** Oracle persistence and loader integration tests persist and
  reload active rule sets using these fields, and database constraints observe
  the validation timestamp. Exact timestamp equality after reload is not asserted
  by the current tests. No claim is made about cross-timezone instant semantics
  because the schema does not provide them.

### Hibernate persistence context and `@OrderBy`

- **Observed failure or risk:** Reading the same managed aggregate immediately
  after saving could make in-memory insertion order look like database ordering.
- **Root cause:** Hibernate's first-level persistence context can return already
  managed collections without reloading them from Oracle.
- **Chosen solution:** Apply `@OrderBy` at stage, rule, condition, and action
  collection boundaries and clear the test persistence context after
  `saveAndFlush` before reloading.
- **Verification:** The integration test deliberately inserts stages, rules, and
  conditions out of order, clears the entity manager, reloads from Oracle, and
  asserts database order.

### Multiple bag-fetch avoidance

- **Observed failure or risk:** One eager join across the aggregate would need to
  fetch multiple ordered `List` collections, notably conditions and actions, and
  risks Hibernate multiple-bag-fetch failures or a large Cartesian result.
- **Root cause:** The rule-definition aggregate contains nested list-valued
  associations at several levels.
- **Chosen solution:** Do not add a broad join-fetch/entity graph. The
  transactional loader navigates the lazy aggregate and maps it completely to
  immutable records before the transaction closes.
- **Verification:** `OracleRuleDefinitionLoaderIntegrationTest` loads the complete
  detached definition successfully, including ordered conditions and actions.
  Query-count and large-rule-set performance were not qualified.

### Activity existence and active-status validation

- **Observed failure or risk:** A rule may have been valid when published but its
  target activity can later be removed or made inactive; adding it would recreate
  a known production defect.
- **Root cause:** Definition-time catalog state is not guaranteed to remain valid
  for the lifetime of an in-memory rule snapshot.
- **Chosen solution:** Validate additions before publication and validate again
  immediately before each stage mutation. If execution-time validation fails,
  leave the failing stage unsaved, retain earlier successful stages, and record an
  operational incident.
- **Verification:** Validator tests reject missing/inactive targets. Action and
  service tests prove no partial stage mutation. Increment 15 changes catalog
  state after publication and verifies the blocked outcome and Oracle incident.

### Plain JUnit versus Spring-backed logging

- **Observed failure or risk:** A malformed-DRL test running as plain JUnit can
  still display Drools' internal compiler logging even though returned exceptions
  and coordinator results are sanitized.
- **Root cause:** `application.properties` logging levels are applied by a Spring
  Boot context; plain JUnit compiler tests do not start that context.
- **Chosen solution:** Production/Spring-backed execution disables the narrow
  internal `KieProject` logger, while code-level diagnostics and refresh results
  are independently sanitized. No test-wide logging override was added merely to
  hide useful plain-test diagnostics.
- **Verification:** Unit tests assert that exception/result data omits sensitive
  input and embedded stack traces. The Spring property verifies configuration
  intent; suppression in every possible external logging configuration remains an
  operational responsibility.

### Failed-refresh last-known-good retention

- **Observed failure or risk:** A malformed replacement version could leave rule
  execution unavailable or expose a partially compiled candidate.
- **Root cause:** Without a publication boundary, load, validation, assembly, or
  compilation failure could mutate shared runtime state.
- **Chosen solution:** Compile candidates off to the side and publish only after
  all phases succeed. Failed phases skip all later phases and do not touch the
  current snapshot.
- **Verification:** A real-KIE refresh test publishes version 17, rejects malformed
  version 18 during compilation, and then executes RA1 through version 17 again.
  Unit tests cover failure at every phase and verify phase ordering and sanitized
  reporting.

### Snapshot retirement while work is in flight

- **Observed failure or risk:** Closing the old runtime immediately on publication
  could break a work order between RA1 and RA2; allowing new readers onto the old
  runtime could prevent retirement indefinitely.
- **Root cause:** Publication and multi-stage execution have overlapping
  lifetimes.
- **Chosen solution:** Atomic publication marks the old slot retired, rejects new
  leases on it, and delays close until its reference count reaches zero. A work
  order holds one lease across RA1, RA2, and RA3.
- **Verification:** Concurrency tests hold readers across publication, prove that
  the old module remains registered while leased, route new readers to the new
  version, and verify old-resource removal after the final lease closes.

## Architectural conclusion

The experiment demonstrates that KIE 10 Rule Units are technically feasible for
the selected rules, including traditional DRL patterns, parameterized rules,
runtime compilation, and the tested traditional `accumulate` form.

It also exposed significant complexity in compilation, dynamic wiring,
classloading, KIE resource ownership, refresh serialization, snapshot lifecycle,
concurrent execution, diagnostic sanitization, startup policy, and operational
control. These are not merely syntax-conversion concerns.

The recommended first production migration is therefore Java 21 and Drools 10
using traditional DRL with conventional KIE sessions. That path preserves the
familiar rules model while separating the platform upgrade from Rule Unit and
dynamic-publication complexity. Rule Units should be reconsidered after the
traditional Drools 10 migration is stable and the operational requirements
identified by this prototype can be designed and qualified deliberately.

This is an architectural recommendation from the experiment, not an implemented
change to the prototype.

## What Increment 15 proves

Increment 15 proves, for the selected synthetic rules and data, that one
application composition can:

- persist and load versioned Oracle rule definitions;
- validate and deterministically assemble traditional DRL;
- runtime-compile and discover three generated Rule Units;
- atomically publish and lease one immutable compiled snapshot;
- run the selected RA1 -> RA2 -> RA3 behavior across per-stage Oracle saves;
- preserve the six agreed characterization contracts through the dynamic path;
- reuse one published version for multiple work orders;
- enforce both definition-time and execution-time activity guards; and
- persist a sanitized operational incident while retaining earlier saved stages
  when a later addition becomes invalid.

It does not prove:

- compatibility with a shared or production-like Oracle environment;
- final Spring bean registration or application startup behavior;
- lazy first-call initialization or the daily/restart loading policy;
- an authenticated and authorized administrative refresh operation;
- production-scale rule volume, work-order throughput, repeated-refresh memory
  retention, or immediate generated-class unloading;
- monitoring, alerting, operational runbooks, deployment rollout, or rollback;
- multi-replica consistency or distributed cache behavior; or
- migration of the complete production ruleset, including every accumulate rule
  and omitted production-only business exception.
